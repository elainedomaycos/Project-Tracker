# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

A lightweight Scrum/task-tracker app ("Task Tracker") built with TanStack Start (React 19, file-based routing), Supabase (Postgres + auth), and Groq (Llama 3.3 70B) for AI-assisted Scrum tooling. It is developed and synced through **Lovable** (lovable.dev) — see the Lovable-managed files note below before editing certain paths.

## Commands

```bash
npm run dev        # start dev server (vite dev)
npm run build      # production build
npm run build:dev  # development-mode build
npm run preview    # preview a production build
npm run lint       # eslint .
npm run format     # prettier --write .
```

There is no test runner configured in this repo.

Package manager: `bun` is configured (`bunfig.toml`) alongside `package-lock.json`/npm — check which lockfile is current before installing. `bunfig.toml` enforces a 24h supply-chain guard (`minimumReleaseAge`) on new package versions; only a short explicit exclude list bypasses it — confirm with the user before adding to that list.

## Architecture

### Routing
TanStack Start uses **file-based routing** rooted at `src/routes/`. Every `.tsx` file there defines a route (see `src/routes/README.md` for naming conventions: `$id` for dynamic segments, `{-$category}` for optional, `$` for splat, `_layout` for layouts). `src/routes/__root.tsx` is the single app shell (nav sidebar, auth gate, providers) — every page renders inside its `<Outlet />`. `routeTree.gen.ts` is auto-generated by the TanStack router plugin; never hand-edit it.

Routes: `index` (dashboard), `tasks`, `developer`, `qa`, `client`, `ai`, `credentials`, `admin`, `auth`.

### Server / client split
- `src/server.ts` — Nitro/Vercel fetch entry point; wraps the TanStack server-entry handler and normalizes h3's swallowed-error responses into a proper 500 error page (`src/lib/error-page.ts`).
- `src/start.ts` — `createStart()` config wiring global `functionMiddleware` (`attachSupabaseAuth`) and `requestMiddleware` (error boundary middleware).
- Files ending in `*.server.ts` are server-only (e.g. `src/lib/ai.server.ts`, `src/integrations/supabase/client.server.ts`). ESLint blocks importing the Next.js `server-only` package — use the `.server.ts` naming convention or `@tanstack/react-start/server-only` instead.
- Server logic (AI calls, privileged Supabase access) is exposed to routes via `createServerFn` (see `src/lib/ai.server.ts`, `src/lib/auth.functions.ts`).

### Supabase integration (auto-generated — do not hand-edit)
`src/integrations/supabase/{client.ts,client.server.ts,auth-middleware.ts,auth-attacher.ts,types.ts}` and `src/integrations/lovable/index.ts` are marked "automatically generated" / "auto-generated by Lovable" and regenerated by the Lovable platform. Avoid manual edits there; if a change is needed, make it in the Lovable project or confirm with the user first.

- `client.ts` — browser Supabase client (anon key, persisted session, `localStorage`).
- `client.server.ts` — service-role Supabase client (`supabaseAdmin`) that bypasses RLS. Server-only; import it dynamically inside server handlers, not at top-level in route or `*.functions.ts` files (those ship to the client bundle).
- `auth-middleware.ts` — `requireSupabaseAuth`, a TanStack server-fn middleware that validates the `Authorization: Bearer` JWT and injects `{ supabase, userId, claims }` into handler context.
- `auth-attacher.ts` — client-side `functionMiddleware` (`attachSupabaseAuth`, registered globally in `src/start.ts`) that attaches the current session's bearer token to every server-fn RPC.

### Two auth/role systems (don't conflate them)
- `src/lib/auth-context.tsx` (`useAuth`) is the **actual app auth**: Supabase session + a `profiles` table row with `role: "super_admin" | "developer" | "qa"`. Super-admin status is derived from a hardcoded `SUPER_ADMIN_EMAILS` allowlist and self-heals the `profiles.role` column on load. New signups get a role from a matching `invitations` row (by email) or default to `developer`; new `developer`/`qa` signups get appended into the `settings` table's `developers`/`qa_users` list.
- `src/lib/project-context.tsx` (`useProject`) additionally tracks a separate, simpler concept: `AppView = "pm" | "developer" | "qa" | "client"` and a `currentDeveloper` PIN-based identity (`PIN_MAP`, `DEFAULT_DEVELOPERS`) — this predates/parallels the Supabase-auth roles and is used for the developer/QA "switch user" UX inside those views, not for route access control.
- Route-level access control in `__root.tsx` (`NAV_ITEMS`/`EXTRA_NAV` + `canSee`) currently allows all three `profile.role` values on every route — role gating is more of a placeholder than an enforced boundary right now.

### Data layer
`ProjectProvider` (`src/lib/project-context.tsx`) loads `projects`, `tasks`, and `settings` (developers/qa_users lists) from Supabase on mount and holds everything in React state; mutators (`addTask`, `updateTask`, etc.) update local state optimistically and fire-and-forget the corresponding Supabase write (`.then(()=>{}).catch(()=>{})` — failures are swallowed, not surfaced to the UI). `toDbTask`/`fromDbTask`/`fromDbProject` convert between camelCase app models and snake_case DB columns.

Two migrations directories exist: `supabase/migrations/` (`00001_schema.sql`, `00002_auth.sql` — original hand-written schema/auth) and `src/supabase/migrations/` (timestamped, Lovable-generated). Check both when tracing schema history; `src/supabase/config.toml` holds the linked `project_id`.

### AI features
`src/lib/ai.server.ts` wraps Groq (`llama-3.3-70b-versatile`) behind several `createServerFn`s (requirement analysis, commit message, branch name, task breakdown, daily/sprint scrum reports, QA check, project insights). Every handler prompts for strict JSON and has a hardcoded mock fallback if `GROQ_API_KEY` is missing or the call/parse fails — check these mocks when a feature "works" locally without a real key configured.

### UI
shadcn/ui (`components.json`, style "new-york") generates components into `src/components/ui/`; Tailwind v4 via `@tailwindcss/vite` (see `src/styles.css`). Path alias `@/*` → `src/*` (both `tsconfig.json` and the Lovable vite config provide this — don't re-declare it).

### Vite config
`vite.config.ts` delegates almost everything to `@lovable.dev/vite-tanstack-config`, which already bundles the TanStack Start/React/Tailwind/tsconfig-paths plugins, Nitro (Vercel preset), dev-only component tagging, env injection, and sandbox port/host detection. Do not add these plugins manually — read the comment at the top of the file before touching it.

## Environment variables

## Lovable sync caveat
This repo is connected to Lovable (see `AGENTS.md`). Avoid rewriting published git history (force-push, rebase/amend/squash of pushed commits) — it desyncs Lovable's view of the project. Keep the connected branch in a working state since every push syncs back into the Lovable editor.
